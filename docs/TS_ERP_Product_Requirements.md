# TS_ERP 产品需求文档

## 1. 文档定位

本文档从产品经理视角定义 TS_ERP 的业务目标、用户角色、功能范围、核心流程、权限边界、数据对象和分阶段路线图。

TS_ERP 按全新系统建设，不做旧系统数据迁移。产品设计从企业真实使用场景出发，重点不是单点 CRUD，而是业务闭环：

- 客户从线索到成交、回款、复购的闭环。
- 商品从主数据、采购、入库、库存、销售、出库、退货的闭环。
- 订单从报价、审批、合同、发货、开票、收款的闭环。
- 财务从应收、应付、付款、收款、对账、利润的闭环。
- 管理者从业务过程、异常预警、经营看板到责任追踪的闭环。

因此 TS_ERP 应从“智能问答 + 基础业务模块”升级为“企业经营管理系统 + 智能助手”。

## 2. 产品目标

### 2.1 总体目标

TS_ERP 面向中小型贸易、分销、轻制造企业，提供一套可代码化管理、可审计、可扩展的 ERP 系统，并通过智能助手提升业务查询、制度问答和运营分析效率。

### 2.2 核心价值

1. 业务流程标准化：让销售、采购、仓库、财务按照统一流程协同。
2. 数据口径统一：客户、商品、库存、订单、财务数据在一个系统中沉淀。
3. 权限和责任清晰：不同岗位看到不同数据，关键操作留痕。
4. 经营状态可视化：管理层可以实时看到销售、库存、回款、利润和异常。
5. 智能助手增强效率：自然语言查询业务数据，基于知识库回答制度流程。

## 3. 目标用户与企业场景

### 3.1 目标企业

- 贸易型企业：多客户、多商品、多仓库、多销售团队。
- 分销型企业：关注采购、库存周转、销售订单、应收回款。
- 轻制造企业：需要基本 BOM、生产计划、领料、完工入库。
- 成长型中小企业：Excel 管理已经失控，但大型 ERP 上线成本过高。

### 3.2 典型岗位

| 岗位 | 关注点 | 高频动作 |
|------|--------|----------|
| 老板 / 总经理 | 销售额、利润、现金流、库存风险 | 看经营看板、查异常、追问原因 |
| 销售经理 | 客户、商机、报价、订单、回款 | 分配客户、审核报价、跟进回款 |
| 销售员 | 客户跟进、报价、下单、发货进度 | 建客户、写跟进、提交订单 |
| 采购员 | 采购需求、供应商、到货、价格 | 询价、下采购单、跟进交付 |
| 仓库管理员 | 入库、出库、库存、盘点 | 收货、拣货、发货、调拨、盘点 |
| 财务人员 | 应收、应付、开票、收付款 | 确认收款、付款、对账、生成报表 |
| 客服 / 售后 | 售后问题、退换货、知识库 | 查订单、查流程、登记售后 |
| 系统管理员 | 用户、角色、权限、基础配置 | 配权限、维护组织、查日志 |

## 4. 产品范围

### 4.1 一期必须覆盖

一期目标是可演示、可试用、能跑通企业核心经营闭环：

- 系统管理：用户、角色、权限、部门、数据权限。
- 基础资料：客户、供应商、商品、仓库、计量单位、币种、结算方式。
- 销售管理：商机、报价、销售订单、发货申请、销售退货。
- 采购管理：采购申请、采购订单、到货入库、采购退货。
- 库存管理：入库、出库、调拨、盘点、库存预警、库存流水。
- 财务管理：应收、应付、收款、付款、对账、毛利统计。
- 智能助手：业务数据查询、制度问答、异常解释、操作指引。
- 报表看板：销售看板、库存看板、财务看板、客户看板。

### 4.2 二期扩展

- 合同管理。
- 发票管理。
- 审批流配置。
- 生产管理：BOM、生产订单、领料、完工入库。
- 多组织、多仓、多账套。
- 移动端轻应用。
- 数据导入导出与批量操作。
- 企业微信 / 钉钉通知。

### 4.3 三期扩展

- MRP 物料需求计划。
- 高级仓储：库位、批次、序列号、波次拣货。
- 成本核算：移动加权、先进先出、批次成本。
- 预算管理。
- 多币种、多税率、多法人。
- 智能经营分析和预测。

## 5. 总体业务流程

### 5.1 销售到收款

```text
线索/客户
  -> 商机跟进
  -> 报价单
  -> 销售订单
  -> 库存检查
  -> 发货出库
  -> 应收生成
  -> 收款登记
  -> 对账完成
```

关键控制点：

- 报价低于最低毛利率时需要审批。
- 销售订单超信用额度时需要审批。
- 库存不足时提示采购或调拨。
- 发货后自动形成应收。
- 收款后更新订单回款状态。

### 5.2 采购到付款

```text
采购需求
  -> 供应商询价
  -> 采购订单
  -> 到货验收
  -> 采购入库
  -> 应付生成
  -> 付款申请
  -> 付款登记
  -> 对账完成
```

关键控制点：

- 采购价格高于历史价格时提示。
- 到货数量与采购订单不一致时记录差异。
- 入库后自动增加库存。
- 付款前校验发票、入库和应付。

### 5.3 库存流转

```text
采购入库 / 销售退货入库 / 盘盈入库
  -> 可用库存
  -> 销售出库 / 采购退货出库 / 调拨出库 / 盘亏出库
  -> 库存流水
```

关键控制点：

- 区分账面库存、可用库存、锁定库存、在途库存。
- 销售订单审核后锁定库存。
- 发货出库后扣减库存。
- 低于安全库存时生成预警。

## 6. 功能模块需求

## 6.1 系统管理

### 6.1.1 组织与部门

功能：

- 维护公司、部门、岗位。
- 支持树形组织结构。
- 用户可归属一个或多个部门。
- 部门可作为数据权限边界。

业务规则：

- 删除部门前必须确认无下级部门和有效用户。
- 部门停用后，不允许分配新用户和新业务单据。

### 6.1.2 用户与角色

功能：

- 用户新增、禁用、重置密码。
- 用户绑定角色、部门、岗位。
- 支持超级管理员、业务管理员、普通用户。

角色建议：

- 超级管理员。
- 总经理。
- 销售经理。
- 销售员。
- 采购员。
- 仓库管理员。
- 财务人员。
- 客服人员。
- 只读审计员。

### 6.1.3 权限体系

权限分为：

- 菜单权限：能看到哪些模块。
- 按钮权限：能新增、编辑、删除、审核、导出。
- API 权限：后端接口鉴权。
- 数据权限：本人、本部门、本部门及下级、全部、自定义。
- 字段权限：敏感字段如成本价、利润、供应商价格可隐藏。

智能助手必须继承当前用户权限，不能绕过业务权限查数据。

## 6.2 基础资料

### 6.2.1 客户档案

字段建议：

- 客户编码、客户名称、客户类型、客户等级。
- 所属行业、客户来源、所在地区。
- 联系人、电话、邮箱、地址。
- 所属销售、所属部门。
- 信用额度、信用账期。
- 开票信息、结算方式。
- 客户状态：潜在、合作中、暂停、黑名单。

功能：

- 客户新增、编辑、停用。
- 客户查重。
- 客户联系人管理。
- 客户跟进记录。
- 客户标签。
- 客户归属转移。
- 客户信用控制。

企业场景：

- 销售创建客户后，销售经理审核。
- 客户超过 90 天未跟进，进入待唤醒客户池。
- 黑名单客户不能创建新订单。

### 6.2.2 供应商档案

字段建议：

- 供应商编码、名称、等级、联系人。
- 供应品类、结算方式、账期。
- 开票信息、银行账户。
- 交付周期、最小起订量。
- 供应商状态。

功能：

- 供应商准入。
- 供应商价格记录。
- 供应商绩效：准时交付率、合格率、价格稳定性。

### 6.2.3 商品档案

字段建议：

- 商品编码、SKU、名称、规格型号、品牌。
- 分类、单位、条码、图片。
- 销售价、成本价、最低售价。
- 税率、重量、体积。
- 是否启用批次、序列号、保质期。
- 安全库存、最高库存、采购提前期。

功能：

- 商品分类管理。
- 多单位换算。
- 价格体系。
- 商品启停用。
- 商品库存策略。

## 6.3 销售管理

### 6.3.1 商机管理

功能：

- 商机登记、阶段推进、预计成交金额。
- 跟进计划和跟进记录。
- 商机来源统计。
- 输单原因分析。

商机阶段：

- 初步接触。
- 需求确认。
- 报价中。
- 谈判中。
- 赢单。
- 输单。

### 6.3.2 报价单

功能：

- 从客户或商机创建报价。
- 选择商品、数量、价格、折扣、税率。
- 支持报价有效期。
- 支持生成 PDF / 打印。
- 支持报价转销售订单。

审批规则：

- 报价低于最低售价需要销售经理审批。
- 毛利率低于阈值需要总经理审批。

### 6.3.3 销售订单

功能：

- 新建订单、编辑订单、审核订单。
- 支持从报价单转订单。
- 自动检查客户信用额度。
- 自动检查库存。
- 审核后锁定库存。
- 支持部分发货、部分收款、订单关闭。

订单状态：

- 草稿。
- 待审核。
- 已审核。
- 待发货。
- 部分发货。
- 已发货。
- 部分收款。
- 已完成。
- 已取消。

### 6.3.4 销售出库与发货

功能：

- 根据销售订单生成发货单。
- 仓库拣货、复核、出库。
- 支持物流单号。
- 出库后扣减库存。
- 自动生成应收。

异常：

- 库存不足。
- 客户超信用额度。
- 发货地址缺失。
- 订单未审核。

### 6.3.5 销售退货

功能：

- 根据订单创建退货申请。
- 售后审核。
- 仓库收货入库。
- 财务冲减应收或退款。

退货原因：

- 质量问题。
- 客户拒收。
- 发错货。
- 价格争议。
- 其他。

## 6.4 采购管理

### 6.4.1 采购需求

来源：

- 手工创建。
- 库存预警触发。
- 销售订单缺货触发。
- 生产计划触发。

功能：

- 采购需求汇总。
- 合并同类商品。
- 选择供应商。
- 转采购订单。

### 6.4.2 采购订单

功能：

- 采购下单。
- 价格、交期、付款条件管理。
- 审核后形成在途库存。
- 支持部分到货。
- 支持关闭未完成数量。

状态：

- 草稿。
- 待审核。
- 已审核。
- 部分到货。
- 已到货。
- 已关闭。

### 6.4.3 采购入库

功能：

- 根据采购订单收货。
- 记录实收数量、拒收数量、差异原因。
- 入库后增加库存。
- 自动生成应付。

### 6.4.4 采购退货

功能：

- 根据采购入库或采购订单创建退货。
- 出库退回供应商。
- 冲减应付或生成供应商退款。

功能：

- 文档上传。
- 文档解析。
- 文档分块。

- 检索日志。

## 6.9 智能助手

### 6.9.1 助手定位

智能助手不是替代 ERP 流程，而是作为企业员工的查询和操作辅助入口。

### 6.9.2 支持问题类型

业务查询：

- “本月销售额是多少？”
- “张三客户还有多少应收未收？”
- “哪些商品库存低于安全库存？”
- “今天有哪些订单待发货？”

制度问答：

- “客户超信用额度时怎么处理？”
- “采购退货需要哪些审批？”
- “库存盘点差异怎么处理？”

混合问题：

- “哪些客户逾期超过 30 天，按照制度下一步怎么跟进？”
- “哪些商品缺货，按采购提前期什么时候必须下单？”

### 6.9.3 智能助手约束

- 必须基于当前登录用户权限回答。
- 业务实时数据必须来自数据库。
- 制度流程必须来自知识库引用。
- 不能编造价格、库存、客户隐私和财务数据。
- 高风险操作只能建议，不能自动执行。

- 毛利分析。
- 客户回款排行。
- 供应商付款排行。

## 8. 审批与通知

### 8.1 审批场景

- 报价低毛利审批。
- 销售订单超信用额度审批。
- 采购订单超预算审批。
- 付款申请审批。
- 盘点差异审批。
- 销售退货审批。

### 8.2 通知场景

- 订单待审核。
- 订单待发货。
- 库存不足。
- 应收逾期。
- 采购到货延期。
- 知识库索引失败。

通知渠道：

- 系统站内信，一期。
- 企业微信 / 钉钉，二期。
- 邮件，二期。

## 9. 数据对象建设建议

数据库按目标业务重新建模。开发阶段可以使用初始化 SQL 和演示数据，但不需要兼容旧系统数据结构。建议建设以下数据对象：

基础资料：

- `suppliers`：供应商。
- `product_categories`：商品分类。
- `units`：计量单位。
- `price_books`：价格体系。

销售：

- `sales_opportunities`：商机。
- `quotations`：报价单。
- `quotation_items`：报价明细。
- `sales_shipments`：销售发货单。
- `sales_returns`：销售退货单。

采购：

- `purchase_requisitions`：采购需求。
- `purchase_orders`：采购订单。
- `purchase_order_items`：采购明细。
- `purchase_receipts`：采购入库单。
- `purchase_returns`：采购退货单。

库存：

- `stock_movements`：库存流水。
- `stock_transfers`：库存调拨。
- `stock_counts`：盘点任务。
- `stock_count_items`：盘点明细。
- `stock_lots`：批次。

财务：

- `receivables`：应收。
- `payables`：应付。
- `receipts`：收款。
- `payments`：付款。
- `invoices`：发票。

审批与审计：

- `approval_flows`：审批流。
- `approval_tasks`：审批任务。
- `operation_logs`：操作日志。
- `chat_logs`：智能助手会话。

## 10. 非功能需求

### 10.1 易用性

- 常用列表支持搜索、筛选、排序、分页。
- 单据支持复制、草稿、提交、撤回。
- 关键字段提供默认值和选择器。
- 数据导入需提供错误行提示。

### 10.2 性能

- 常用列表 2 秒内返回。
- 首页看板可使用缓存。
- 大数据导出走异步任务。
- 智能助手查询需要展示执行状态。

### 10.3 安全

- 所有业务接口需要认证。
- 敏感字段按权限显示。
- 删除采用逻辑删除优先。
- 高风险操作记录操作日志。
- 动态 SQL 必须经过白名单和只读校验。

### 10.4 可审计

- 记录谁在什么时候改了什么。
- 关键单据保存状态流转记录。
- 智能助手保存问题、意图、引用、SQL、结果摘要。

## 11. 分阶段路线图

### 11.1 M0：体系确认

- 明确 Spring Boot 作为主业务后端。
- Python 保留为 Agent/RAG 服务。
- 前端继续 Vue 3。
- 明确三项目边界：`frontend/`、`java-backend/`、`agent_service/`。
- MySQL、Redis、ES 使用统一配置入口。

### 11.2 M1：企业基础可用版

目标：跑通销售、库存、财务最小闭环。

## 12. 验收标准

一期验收必须满足：

1. 销售订单可以从创建到审核、出库、形成应收。
2. 采购订单可以从创建到审核、入库、形成应付。
3. 库存可以查询现存量、可用量和库存流水。
4. 财务可以登记收款、付款，并查看应收应付。
5. 管理者可以看到销售、库存、财务核心看板。
6. 不同角色看到的数据和菜单不同。
7. 智能助手可以回答权限范围内的业务查询。
8. 智能助手回答制度问题时能引用知识库来源。
9. 关键操作有日志。
10. 系统配置 MySQL、Redis、ES host 和端口后可本机启动。
11. 前端只调用 Java 后端，Java 后端统一调用 Python Agent Service。

## 13. 参考产品能力

本产品设计参考了成熟 ERP 的公开能力边界：

- SAP Business One：中小企业一体化 ERP，覆盖财务、销售、采购、库存、生产和分析。
- Microsoft Dynamics 365 Business Central：覆盖财务、销售、采购、库存、仓储、项目、服务和制造等业务功能。
- Oracle NetSuite ERP：覆盖财务、订单、库存、采购、CRM、仓储和供应链。
- Odoo：以模块化应用覆盖 CRM、销售、采购、库存、会计、制造、项目和电商等场景。

TS_ERP 不直接照搬大型 ERP，而是优先选取中小企业最常用、最能形成业务闭环的能力，并结合智能助手形成差异化。
